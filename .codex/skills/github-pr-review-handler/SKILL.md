# GitHub PR Review Handler

レビュー指摘の対応可否を判断してから修正する。未解決スレッドをGitHub MCPで洗い出し、PRの差分と周辺コードを確認し、必要な修正とレビュー返信を一貫して行う。

レビュー対応だけを目的とする作業では、`docs/todo.md` の確認や更新を不要として扱う。

## 入力

- 必須: PR番号
- 推奨: レビュー方針に関する追加指示
  - 例: 「破壊的変更は避ける」「テスト追加を優先する」

## 基本方針

- レビュー指摘をそのまま受け入れない。まず事実確認と意図の再構成を行う。
- 指摘箇所だけで判断しない。必ずこのPRの差分と前後の実装文脈を見る。
- 返信は、結論、判断理由、実施内容を短く具体的に書く。
- すでに満たしている指摘や採用しない指摘にも、理由を添えて返信する。
- 返信前に、修正内容とコメント内容が矛盾していないことを確認する。

## ワークフロー

### 1. PRの前提を集める

- GitHub MCPでPRの概要を取得する。
  - `mcp__github__pull_request_read` の `get`
- 変更ファイル一覧を取得する。
  - `mcp__github__pull_request_read` の `get_files`
- 必要に応じてPR全体のdiffを確認する。
  - `mcp__github__pull_request_read` の `get_diff`
- ローカルリポジトリでは、必ずベースブランチとの差分でも確認する。
  - 例: `git diff <base>...HEAD -- <path>`
  - 例: `git show <base>:<path>`

### 2. 未解決レビューを洗い出す

- GitHub MCPでレビューコメントスレッドを取得する。
  - `mcp__github__pull_request_read` の `get_review_comments`
- `isResolved` が `false` のスレッドだけを対象にする。
- 各スレッドについて次を整理する。
  - 対象ファイル
  - 対象行
  - 元の指摘内容
  - 最新コメント
  - 返信先に使う `commentId`
- 未解決スレッドが0件なら、その旨を報告して終了する。

### 3. 指摘の妥当性を検証する

各指摘で次を順に確認する。

1. 指摘されたコードは、このPRで何を目的に変更した箇所か
2. 指摘は事実として正しいか
3. 指摘を採用すると、このPRの目的や既存設計と矛盾しないか
4. すでに別の変更で解消済みではないか
5. 追加修正が必要なら、最小変更で解決できるか

判定は次の3分類で扱う。

- `対応する`
  - 指摘が妥当で、PRの目的と整合する
- `対応不要`
  - 誤認、すでに対応済み、あるいは今回のPR範囲外
- `要確認`
  - 仕様判断や設計判断が必要で、ローカル情報だけでは決められない

`要確認` に該当したら、憶測で進めずユーザーに確認する。

### 4. 必要な修正を行う

- `対応する` と判断した指摘だけを修正する。
- 修正はレビュー指摘を満たす最小単位に留める。
- 影響範囲がある場合は関連テストも更新する。
- 実装後は関連テストや必要最小限の検証を実行する。

### 5. レビューへ返信する

- 修正が発生した場合は、返信前に変更ファイルをステージしてコミットし、現在の作業ブランチへ push する。
- コミットはリポジトリの規約に従い、変更対象ファイルだけを `git add <path>` で追加する。
- 修正が発生しなかった場合は、コミットや push は不要。
- 返信には `mcp__github__add_reply_to_pull_request_comment` を使う。
- 返信先は、そのスレッド内で解決対象として最も適切な `commentId` を選ぶ。
- 返信は日本語で簡潔に書く。
- 修正した場合は、何を直したかを具体的に書く。
- 対応不要の場合は、採用しない理由を簡潔に説明する。

返信テンプレート:

- 対応した場合
  - `ご指摘ありがとうございます。<理由>のため、<対応内容>を反映しました。`
- 対応不要の場合
  - `ご指摘ありがとうございます。<確認内容>を確認しましたが、<理由>のため今回は対応を見送ります。`
- すでに解消済みの場合
  - `ご指摘ありがとうございます。<どの変更>で既に解消されていることを確認しました。`

### 6. 結果を報告する

- 最後にユーザーへ次をまとめる。
  - 未解決レビュー件数
  - 各指摘の判定結果
  - 実施した修正
  - 実行した検証
  - 作成したコミットと push の有無
  - 返信したコメントの要旨

## 実行上の注意

- PRのレビュー対応では、未解決スレッドを優先し、解決済みスレッドは原則追わない。
- 返信前に、対象スレッドがすでに他者の返信で実質解決していないか確認する。
- 大きな設計変更が必要なら、その場で広げずユーザー確認に切り替える。
- GitHub関連操作はGitHub MCPを優先し、`gh` CLIは使わない。
