# Memora Dev Workflow

このスキルは Memora でコード変更を行うときの進め方だけを定義する。規約、制約、命名、アーキテクチャ方針、MCP使用ルールは `AGENTS.md` を優先して参照する。

## 開始手順

1. `docs/todo.md` に該当作業があるか確認する。
2. `todo` 確認を省略する指示がない限り、該当項目がなければ作業を止めてユーザーに確認する。
3. 現在のブランチが `main` の場合だけ、新しい作業ブランチを切る。
4. `tree lib test` を実行し、対象機能の配置を把握する。
5. 実装前に必要な設計資料を確認する。
6. 実装中のルールと制約は `AGENTS.md` を参照する。

設計資料:
- `docs/app_spec.md`
- `docs/user_stories.md`
- `docs/usecase_diagram.md`
- `docs/er_diagram.md`
- `docs/todo.md`
- `docs/done.md`

## 実装手順

常に Red -> Green -> Refactor の順で進める。

### 1. Red

- まず失敗する小さなテストを書く。
- そのテストだけを実行し、期待どおりに失敗することを確認する。
- テストが失敗する前に実装コードを書かない。
- 失敗理由が想定とずれている場合は、先にテストを修正する。

### 2. Green

- 失敗中のテストを通す最小限の実装だけを書く。
- 余計な一般化や最適化を入れない。
- テストを再実行し、Green になることを確認する。

### 3. Refactor

- 重複を取り除き、構造を整える。
- 機能追加はせず、既存の振る舞いを保ったまま整理する。
- テストを再実行し、通り続けることを確認する。

判断基準:
- 大きな変更を一度に入れず、小さなステップに分割する。
- まずは定数を返す仮実装から始めてよい。
- 必要に応じて三角測量で一般化する。
- 実装が明白なら直接書いてよいが、Red を飛ばさない。

## 完了手順

1. `./check.sh` を実行し、フォーマット・解析・テストが通ることを確認する。
2. 対応した `docs/todo.md` の項目を `docs/done.md` へ移動する。
3. 関連ファイルだけを `git add <対象ファイル>` でステージングする。`git add .` は使わない。
4. Conventional Commits 形式でコミットする。
5. ブランチを push する。
6. GitHub MCP でプルリクエストを作成する。
7. PR本文には `docs/done.md` に移した作業項目を記載する。
