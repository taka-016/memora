# Coding

実装・修正・リファクタリングを行うときの実行手順を定義する。
このスキルの目的は、TDD と設計制約を「理解している状態」ではなく「実際に守る状態」にすること。

## 基本方針

- TDDのRed→Green→Refactorのサイクルで進める
- AIの判断で手順を省略したり完了条件を緩和することは禁止
  - 本リポジトリのオーナーは、手順と完了条件に従わない作業を必ず破棄する
  - token節約のために条件の省略や完了条件を緩和する行為は、逆にtokenを無駄遣いする結果になると理解しなさい
- `test-coding`スキルを必ず併用する
- 各フェーズの完了条件を満たすまで次のフェーズへ進んではならない

## 手順

### 1. Red

- 新しい振る舞い、または修正対象を表す小さなテストを1つだけ追加する
- 対象テストを実行し、期待どおりに失敗することを確認する
- 失敗理由が想定外なら、実装ではなくテスト設計を見直す
- この時点では実装コードを編集しない
- Red完了後は必ずコミットする

### 2. Green

- Redで追加したテストを通すための最小限の実装だけを書く
- 対象テストを実行し、成功することを確認する
- Green中にテストコードを修正する必要が出たら、その変更を破棄してRedに戻る
- 実装後に差分を確認し、意図せずテストを変えていないことを確認する
- Green完了後は必ずコミットする

### 3. Refactor

- 重複排除と設計改善だけを行う
- 必要なら実装コードとテストコードの両方を整える
- リファクタリング後に対象テスト、必要なら関連テストを再実行する
- 新しい振る舞いを足したくなったら、その場で止めて次のRedとして切り出す
- Refactor完了後は必ずコミットする

### 4. サイクル継続判断

- まだ要求が残る場合は、次のRedを1件だけ追加してサイクルを繰り返す
- 実装が一段落したら、関連テスト全体と必要な検証へ進む

## フェーズ完了条件

### Red完了条件

- 新しい失敗テストが1件ある
- そのテストの失敗理由が期待どおりである
- 実装コードはまだ触っていない
- コミットが完了している

### Green完了条件

- 直前のRedで追加したテストが通る
- 実装はそのテストを通す最小限に留まっている
- Green中にテスト仕様を書き換えていない
- コミットが完了している

### Refactor完了条件

- 外部振る舞いは変わっていない
- 対象テストが通る
- 次のRedに進むか、完了検証に進める状態になっている
- コミットが完了している

## TDDの三原則

1. 失敗するユニットテストを書くまでは、実装コードを書いてはならない
2. コンパイルが通らず失敗するユニットテストを書く以上に、ユニットテストを書いてはならない
3. 現在失敗しているテストを通す以上に、実装コードを書いてはならない

## TDDで重要な考え方

- 小さなステップ: 大きな問題を小さく分割して進める
- 仮実装: まずは定数を返すなど、最小限の実装から始める
- 三角測量: 複数のテストケースから一般化を行う
- 明白な実装: 実装が自明な場合は直接書く
