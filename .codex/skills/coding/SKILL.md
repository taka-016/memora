# Coding

実装中は、設計制約を守りつつ小さなTDDサイクルで進める。

## 基本ルール

- コメントは、コードを見ても意図が読み取れない場合にのみ記載する。
- プレゼンテーション層は`application/usecases`を参照し、必要に応じて`application/dtos`も参照してよい。`domain/*`と`infrastructure/*`を直接参照しない。
- 作業中に仕様判断や設計判断が必要になった場合は、憶測で進めず作業を中断してユーザーに確認する。

## TDDワークフロー

Kent Beckの原著『Test-Driven Development: By Example』と、その翻訳者であるt-wadaの解釈に従って進める。

### Red

- 新しい機能や修正に対する小さな失敗テストを書く。
- テストを実行し、期待通りに失敗することを確認する。
- この段階では実装コードを書かない。

### Green

- 失敗中のテストを通すための最小限の実装を書く。
- テストを再実行し、成功することを確認する。
- 設計を広げず、まずは通すことに集中する。

### Refactor

- テストが通った状態を維持したまま重複を排除し、設計を改善する。
- リファクタリング後は必ず再度テストを実行する。
- 機能追加とリファクタリングを同時に進めない。

## TDDの三原則

1. 失敗するユニットテストを書くまでは、実装コードを書かない。
2. コンパイルが通らず失敗するユニットテストを書く以上に、ユニットテストを書かない。
3. 現在失敗しているテストを通す以上に、実装コードを書かない。

## リファクタリング時の追加ルール

- リファクタリングを行う際は、`AGENTS.md`の「リファクタリング定義」に従う。
- 外部振る舞いを変えず、小さなステップで進める。
- テストで安全性を確保できる状態でのみ行う。
- 包括的なテストスイートがないまま大きな整理を始めない。
