# PR Implementation Explainer

PRの差分を実装単位で読み解き、対象箇所のコードを短く示しながら順番に説明する。差分の列挙ではなく、「このPRで何をどう変えたか」が流れで分かる説明を作る。

## 入力

- 必須: 対象PRを特定できる情報
  - 最短ならPR番号だけでよい
  - 例: PR番号、比較対象ブランチ、現在の作業ブランチ
- 任意: 説明方針
  - 例: 「TopPageを重点的に」「初学者向けに」「UI変更は省略」

最小の起動例:

- `PR番号123の対応を解説して`
- `PR 123を解説して`
- `このブランチの実装内容を解説して`

## 基本方針

- テストコード、モック、`docs` の説明は省く。明示依頼がある場合のみ扱う。ただし `docs/todo.md` は説明対象ではなく、PRの背景把握のための確認対象として先に見る。
- 変更ファイルをそのまま列挙せず、実装の責務や処理の流れでまとめる。
- コード差分に入る前に、`docs/todo.md` の差分で完了タスクを把握し、PR内のレビューやコメントのやりとりで論点を確認する。todo作成を含めてPRした後、タスク完了に伴ってtodoを削除している場合は、最終差分上で `docs/todo.md` の差分が見えないことがある。その場合は、PRに紐づくコミットログからtodo作成や更新に関連するコミットを確認する。
- 必ず差分と変更後ソースの両方を見る。差分だけで説明しない。
- コード引用は必要最小限に絞り、引用の直後にその役割を説明する。
- ユーザーが理解したいのは「何が増えたか」より「なぜこの形で実装したか」であることを意識する。

## ワークフロー

### 1. PRの背景を先に把握する

- コード差分を見る前に、`docs/todo.md` の差分を確認して、このPRで完了扱いにしたタスクを把握する。
- `docs/todo.md` の差分がない、または完了タスクを判断できない場合は、PRに紐づくコミットログを確認し、todo作成・todo更新に関連するコミットからPRの背景タスクを把握する。
- PR番号がある場合は、GitHub MCPでPRのレビュー、レビューコメント、会話コメントを確認し、指摘への対応や設計判断の経緯を押さえる。
- 先に把握した内容をもとに、「このPRは何を完了させる変更か」を短く言語化する。

### 2. 対象PRの差分を特定する

- PR番号がある場合は、GitHub MCPでPRの概要と変更ファイルを確認する。
- ローカルに対象ブランチがある場合は、ベースブランチとの差分も確認する。
  - 例: `git diff <base>...HEAD --stat`
  - 例: `git diff <base>...HEAD -- <path>`
- 変更ファイルのうち、実装説明の対象外を外す。
  - `test/`
  - `docs/` ただし `docs/todo.md` は背景把握に使うため除外しない
  - 自動生成ファイル

### 3. 実装の主題を整理する

- 残った差分を見て、「このPRが何を達成する変更か」を1文で要約する。
- その要約をもとに、説明する単位を決める。
  - 入口処理
  - 状態管理
  - 表示部品
  - 既存責務の移動
- ファイル単位ではなく、振る舞い単位で分ける。

### 4. 対象箇所のソースを読む

- 各説明単位について、変更箇所の前後も含めて読む。
- 必要なら次を確認する。
  - 呼び出し元
  - 呼び出し先
  - 状態の保持場所
  - 分岐条件の意味
- 説明に使うコードは、要点が分かる最小の断片だけ抜き出す。

### 5. 実装内容を1つずつ解説する

各項目は次の順で説明する。

1. どの責務の変更か
2. 対象ファイル
3. 短いコード断片
4. このコードが何をしているか
5. PR全体の中でなぜ必要か

説明テンプレート:

1. `<変更テーマ>`  
   対象: `[file.dart](/abs/path/file.dart:line)`

   ```dart
   // 変更の要点が分かる最小コード
   ```

   ここでは `<処理内容>` を追加・変更している。  
   これにより `<振る舞いの変化>` が起きる。  
   このPR全体では `<他の変更とのつながり>` を担っている。

### 6. 最後に全体の流れを短くまとめる

- 実装項目を説明し終えたら、最後に1回だけ全体の流れをまとめる。
- まとめは3行程度で十分。
- 例:
  - `TopPageで入口を検知`
  - `notifierで分岐を判断`
  - `一覧または詳細表示へつなぐ`

## 出力ルール

- 見出しは「実装の流れ」または「変更点」のような短いものにする。
- 1項目ずつ番号付きで説明する。
- 各項目には必ず対象ファイルのリンクを付ける。
- コード引用は短く保ち、長い関数全文は貼らない。
- テストの解説は入れない。
- 差分の網羅説明ではなく、読み手が追いやすい順に並べ替えてよい。

## 実行上の注意

- 変更箇所が多い場合でも、同じ目的の変更は1項目にまとめる。
- `docs/todo.md` とPR上のやりとりで把握したタスクや論点から外れた説明になっていないかを確認する。
- 変更後コードだけでは意図が分からない場合は、差分や旧実装も確認した上で説明する。
- 推測が入る場合は断定せず、「この構成にしている意図は〜と考えられる」と明示する。
- ユーザーが特定箇所を追加で質問しやすいよう、最初の回答では深掘りしすぎない。
